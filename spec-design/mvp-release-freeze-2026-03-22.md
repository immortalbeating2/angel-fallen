# MVP 发布冻结清单（2026-03-22）

## 目标

- 将项目从“代码/测试基础设施持续扩张”切换到“可发布 MVP 收口”。
- 冻结范围以 `spec-design/roadmap.md` 的 MVP 目标为准，但允许对实现形态做务实收敛。
- 本清单优先回答两个问题：
  - 什么内容已经足够进入 MVP 发布范围。
  - 什么缺口仍然阻塞“可发布”，必须在继续 D44+ 之前补齐。

## 当前基线（D43 后）

- 代码、测试、配置文档已追平到 D43：`chapter_snapshot_v19` + `multi_cycle_adaptive_gate` + `release_feedback_governance`。
- 主循环已具备可玩闭环：角色选择 -> 开局 -> 房间推进 -> 商店/事件/Boss -> 结算落盘。
- Meta、角色选择、结算、章节转场、记忆碎片、Codex、运行时设置、输入重映射均已有实现入口。
- CI 已覆盖 `ubuntu-latest` + `windows-latest` 的 headless + GUT + quality/release/compatibility/resource/visual snapshot 链路。
- Windows 本地已使用 Godot headless 实测跑通全量 GUT：`52 tests / 6217 assertions`。

## MVP 冻结范围

| 模块 | 冻结结论 | 当前判断 | MVP 发布要求 |
|---|---|---|---|
| 核心战斗循环 | 保留 | 已满足 | 玩家移动/冲刺/自动攻击/敌人压力/升级/房间推进保持现状，不再做大改。 |
| Roguelike 探索 | 保留 | 已满足 | Run-plan 地图、房间类型、Boss 房推进保持可通关闭环。 |
| 内容范围 | 冻结为 MVP 最小集 | 基本满足 | 以 `2章(F1-F6) + 2 Boss + 3角色` 为发布目标；现有超出部分视为 bonus，不再继续扩规格。 |
| 角色系统 | 保留 | 已满足 | 至少稳定支持骑士/法师/盗贼开局、存档记忆上次选择、角色差异可感知。 |
| Meta 进度 | 保留 | 已满足基础闭环 | 保证 run result -> meta currency -> meta upgrade -> 下一局生效闭环稳定；UI 形态可沿用当前主菜单实现。 |
| 叙事与章节转场 | 保留 | 已满足基础闭环 | 保持章节介绍/转场/事件选择/记忆碎片链路可用，不要求先拆成独立 `narrative/` 场景树。 |
| 安全营地/锻造/记忆祭坛 | 降级冻结 | 已有嵌入式实现 | MVP 接受嵌入在 `game_world`/现有 UI 中的交互，不以单独 `safe_camp.tscn` 作为发布前置条件。 |
| 存档系统 | 降级冻结 | 部分满足 | MVP 只要求持久化、损坏回退、基本稳定；`加密 + 多备份 + 迁移框架` 延后到发布后或 v1.1。 |
| 音频与表现 | 降级冻结 | 部分满足 | 先保证“有反馈、不刺耳、可调节”，不以完整 crossfade/SFX 池、全量正式素材作为发布阻塞项。 |
| QA/回归链 | 保留 | 已很强 | 继续把现有 Python 校验、GUT、quality/release/compatibility/visual snapshot 作为发布前主门禁。 |

## 明确延期到 MVP 之后的事项

- 文档中的完整目录形态（如独立 `safe_camp/`、更完整的 `narrative/` 场景树）不作为 MVP 发布阻塞。
- 存档系统的完整规格版本（加密、备份、迁移框架）不作为当前 MVP 发布阻塞，但应列入 v1.1 硬化计划。
- macOS CI 自动化矩阵可暂缓；但 macOS 导出与人工 smoke 不能缺失。
- 全量正式美术/音频替换不作为当前阻塞项；允许带占位/半成品资源进入内部或小范围 MVP 发布。
- 本地化、无障碍、手柄深度适配按 `P1/P2` 处理，不阻塞最短发布路径。
- D44 及后续视觉治理扩展暂停，直到发布闭环最小集合补齐。

## P0 发布阻塞项

### 1. 三平台导出已打通，仍缺跨平台人工验收闭环

- 现状：已补齐 `Windows Desktop`、`Linux/X11`、`macOS` 三个平台导出预设，并已安装 `4.6.1.stable` export templates；当前已成功生成首轮三平台产物：`builds/windows/angel-fallen.exe`、`builds/windows/angel-fallen.pck`、`builds/linux/angel-fallen.x86_64`、`builds/linux/angel-fallen.pck`、`builds/macos/angel-fallen.zip`，且已补齐文件大小与 SHA256 归档。
- 阻塞原因：路线图的 MVP 发布目标明确包含 `Win/Linux/Mac`。
- 发布前最低要求：
  - 为 3 个平台补齐执行时间、执行人、问题单链接与人工验收结论。
  - 持续维护基础导出信息（导出路径、版本、产品名、图标占位）。
  - 每个平台至少保留 1 份可追溯导出记录，并在后续发布迭代中可重复复现。

### 2. 发布 smoke 清单已固化，但首轮跨平台执行仍未完成

- 现状：已新增 `release-smoke-checklist-2026-03-22.md`，并已有首轮自动化/导出演练记录；当前已具备实际导出产物，并已完成 Windows 主机最小启动 smoke，但尚未完成基于 Linux / macOS 目标环境的人工 smoke 记录。
- 阻塞原因：自动化通过不等于发行前体验通过。
- 发布前最低要求：
  - 按清单完成至少 1 轮完整执行并存档结果。
  - 至少覆盖：启动、角色选择、首局开始、升级、商店、Boss、结算、返回主菜单、存档重开、设置修改持久化。

### 3. 首轮完整 MVP 通关验证未成文

- 现状：已有集成测试覆盖关键流程，但尚未形成“2章(F1-F6) + 2 Boss”完整人工/录像/记录闭环。
- 阻塞原因：可测试不等于可发布，必须确认完整体验链不在中后段断裂。
- 发布前最低要求：
  - 完成至少 1 次从开局到第二 Boss 的完整人工通关记录。
  - 覆盖 victory / retreat / death 三种结算分支中的至少前两种，推荐三种都留记录。

### 4. 新手引导/首局提示不足

- 现状：路线图要求 `F1 五步教程`，仓库里尚未看到独立教程实现证据。
- 阻塞原因：公开 MVP 发布若没有最小引导，首局流失风险过高。
- 发布前最低要求：
  - 不要求完整教程系统。
  - 但至少需要首局控制提示 + 房间推进提示 + 升级提示 + 商店提示 + Boss 提示的最小版本。

### 5. 发布包元数据与流程文档缺失

- 现状：导出预设已补入基础版本号、产品名、公司名、bundle identifier 与导出路径，但仍缺少发布归档模板、校验值记录规范与回滚说明。
- 阻塞原因：即使能导出，也还不是“可交付发布包”。
- 发布前最低要求：
  - 维持版本号、产品名、基础描述、图标占位的一致性。
  - 明确构建命名规则、产物目录、校验值记录方式、回滚方式。

## P1 收口项（不阻塞最短发布路径，但建议紧随其后）

- 存档硬化：至少增加一个 backup copy 或 version 字段，为后续迁移做准备。
- AudioManager 体验升级：BGM 切换、SFX 峰值、音量平衡。
- 性能验证补强：聚焦 `500 敌 >= 55 FPS` 的真实压力确认。
- 场景结构补正：把当前集中在 `game_world.gd` 的营地/叙事能力逐步拆分。
- 正式素材替换：优先替换主菜单、角色、Boss、关键 UI 资源。

## 最短执行顺序

1. 基于现有 Windows / Linux / macOS 产物执行首轮人工 smoke，并归档通过项与失败项。
2. 进行 1 轮完整 MVP 人工通关验证，记录问题单。
3. 以“最小提示”方式补一个首局引导层，不先做大教程系统。
4. 固化产物归档、校验值记录与回滚说明。
5. 清零 P0 阻塞项后，再进入性能/音频/素材收口。

## 本清单的执行约束

- 从本清单开始，新增工作默认以“是否缩短到 MVP 发布”为判断标准。
- 若一个任务不能直接提升发布闭环、体验闭环或稳定性闭环，默认后移。
- D44+ 治理型扩展暂停，直到本清单中的 P0 项全部转为完成。
